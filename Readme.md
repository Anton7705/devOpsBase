### **Часть 1**

В этом проекте представлены два Dockerfile для Java-приложения, 
демонстрирующие плохие и хорошие практики контейнеризации. 
Оба файла выполняют одну и ту же задачу — запускают простое Java-приложение, 
выводящее "Hello". Однако их реализация отличается с точки зрения 
безопасности, эффективности и поддерживаемости.

**1. Использование JDK вместо JRE и тяжелого базового образа**

Почему плохо:
В BAD Dockerfile используется eclipse-temurin:21-jdk. 
JDK много ненужного для запуска готового приложения. 
Это увеличивает размер образа на +- 300 МБ.

Решение:
Использован легковесный образ jre-alpine (только среда выполнения Java) с указанием конкретной версии. Размер меньше, меньше лишних зависимостей, ускорена загрузка.

**2. Запуск приложения от имени root**

Почему плохо:
В BAD Dockerfile отсутствует создание пользователя, и контейнер запускается от root.
Это дает процессам внутри контейнера полный доступ к файловой системе и ресурсам с правами суперпользователя. 
Если в приложении есть уязвимость, при атаке можно получить контроль над контейнером и использовать его для атаки на хост-систему и другие контейнеры.

Решение:
Создан системный пользователь appuser и группа appgroup, приложение запускается от этого пользователя. Ограничение прав при взломе системы.

**3. Использование ADD вместо COPY**

Почему плохо:
ADD имеет неочевидное поведение: она автоматически распаковывает архивы и может скачивать файлы по URL.
Если цель — просто скопировать локальный JAR-файл, то ADD может сработать непредсказуемо (например, при совпадении имени с архивом).

Решение:
Использована директива COPY, которая выполняет только одну задачу — копирование файлов из контекста сборки.
Более предсказуемое поведение, меньше рисков.

**4. Относительный путь WORKDIR**

Почему плохо:
Непонятно, где именно находится рабочая директория, и если предыдущий слой оставил какой-то контекст, путь может измениться.

Решение:
Использован абсолютный путь /app. Файлы проекта изолированы в своей директории. Явное указание местоположения упрощает навигацию внутри контейнера.

**5. Отсутствие абсолютного пути в ENTRYPOINT**

Почему плохо:
Если WORKDIR изменится или контекст будет другим, JAR-файл может не найтись.

Решение:
Указан абсолютный путь к JAR-файлу. Контейнер всегда найдет JAR-файл независимо от контекста.

### **Плохие практики работы с контейнерами**

1. Плохая практика работы с контейнером - отсутствие доп изоляции. Дефолтно контейнеры в одной пользовательской сети могут общаться друг с другом,
что не всегда хорошо. + Можно пробросить уровень изоляции 0.0.0.0 что будет означать, что контейнер становится доступным для всех — с хоста, из локальной сети и из интернета, что увеличивает риск атак.


2. Дефолтно контейнер может потреблять все ресурсы хоста (CPU, память, дисковое пространство), что не всегда хорошо. Если один контейнер начнет 
бесконтрольно использовать ресурсы, он может "уронить" хост-машину и все остальные контейнеры, работающие на ней. Это приводит к эффекту "соседа-шума" (noisy neighbor), когда одно приложение мешает работе других.

## 2 ЧАСТЬ

**1. Использование тега latest для образов**

Почему плохо:
В BAD Docker Compose используется postgres:latest и hello-app:latest. Тег latest не гарантирует консистентность — при каждом запуске может подтянуться новая версия образа.
Это приводит к недетерминированным сборкам.

Решение:
Использованы фиксированные версии образов: postgres:16.1 и hello-app:good. Это гарантирует, что всегда будет использоваться одна и та же версия.


**2. Отсутствие healthcheck для базы данных**

Почему плохо:
В BAD Docker Compose отсутствует healthcheck для сервиса db. Контейнер app использует depends_on: - db, что означает зависимость только по факту запуска контейнера, а не по его готовности принимать запросы.
PostgreSQL может стартовать 5-10 секунд, но приложение запускается сразу и получает ошибки подключения. Это приводит к падению приложения при каждом старте (или вообще рандомно, когда как, в зависимости от скорости запуска постгрес).

Решение:
Добавлен healthcheck для PostgreSQL с проверкой pg_isready. 
В depends_on указано условие condition: service_healthy. Приложение запускается только после того, как БД действительно готова принимать соединения.

**3. Отсутствие отдельного сервиса для инициализации базы данных**

Почему плохо:
В BAD Docker Compose отсутствует сервис db-init. Таблицы не создаются автоматически при запуске. 
Приходится вручную заходить в контейнер БД и выполнять SQL-скрипты (или это будет делать каждый раз приложение, что тоже вариант. НО у приложоения может не быть прав на изменение схемы БД + инициализация в db_init происходит только 1 раз).

Решение:
Добавлен отдельный сервис db-init, который использует тот же образ PostgreSQL, зависит от здоровой БД и выполняет SQL-скрипты для создания таблиц. Сервис запускается один раз, создает структуру и завершается.

### Изоляция

Сделал 2 отдельные сети
`db-network:
driver: bridge
app-network:
driver: bridge`

Отнес контейнеры к разным сетям. Контейнеры в разных сетях не видят друг друга, могут общаться через порты хост машины
